# 6.6.3 — UX Integrity Tests
## Specifica architetturale concettuale formale (SPEC-ONLY)

**Microstep:** 6.6.3 — UX Integrity Tests  
**Collocazione:** Fase 6 — Event, Behavior & Value Layer (Sistema IRIS)  
**Blocco:** 6.6 — UX Safety Net  
**Prerequisiti:** 6.6.1 (Futile Chat Detection), 6.6.2 (Silence-First Defaults) — specifiche di riferimento  
**Natura:** Guardrail di validazione · Non-feature · Non-runtime

**Contesto architetturale (BLOCCO 6.6):** Il BLOCCO 6.6 — UX Safety Net garantisce che IRIS **non parli quando non deve**, **non analizzi quando non è appropriato**, **non produca output UX non autorizzato**. Il microstep 6.6.3 definisce **requisiti architetturali verificabili** per garantire che IRIS non produca output UX non autorizzato, né intenzionale né accidentale.

UX Integrity Tests **non** sono una feature né un servizio runtime; sono un **meccanismo di enforcement architetturale** che verifica, in modo sistematico e ripetibile, il rispetto del silenzio di default e l’assenza di regressioni percettive.

## 2. Scopo del microstep

Lo scopo di **UX Integrity Tests** è **garantire**, in modo **sistematico** e **ripetibile**, che:

- **IRIS rimanga silenzioso per default** — in assenza di autorizzazione esplicita, **nessun** output **MUST** essere prodotto
- **Nessun layer, feature o regressione introduca** output UX accidentali o regressioni percettive, in particolare:
  - **output UX** — messaggi, notifiche, schermate non autorizzate
  - **suggerimenti** — hint, raccomandazioni, prompt impliciti non autorizzati
  - **segnali impliciti** — indicatori, badge, micro-feedback non autorizzati
  - **comportamenti percettivamente attivi** — timing, pause comunicative, cambiamenti osservabili non autorizzati

Questo microstep **non** introduce funzionalità; **protegge** la **coerenza percettiva** del sistema nel tempo e la **non-intrusività** come base della fiducia utente. I test di integrità UX sono un **guardrail sistemico** che **MUST** essere rispettato da ogni evoluzione del sistema.

---

## 3. Principio fondamentale: Percettive Invariance

**Definizione formale del principio:**

> Se nessuna feature è autorizzata a produrre output, l’esperienza percepita **MUST** essere identica a quella di un sistema senza IRIS.

**Conseguenze vincolanti:**

- I test di integrità UX **verificano l’assenza di differenza percettiva**, **non** la presenza di funzionalità. L’obiettivo **MUST** essere: **nessun** output osservabile senza autorizzazione; **nessuna** differenza percettiva tra “IRIS attivo ma silente” e “IRIS assente o disabilitato”.
- **Il silenzio è uno stato verificabile** — l’assenza di output **MUST** essere verificabile in modo sistematico e ripetibile; i test di integrità UX definiscono i criteri architetturali per tale verifica, **senza** prescrivere strumenti né implementazione.

---

## 4. Ambito di applicazione

I test di integrità UX **coprono** (senza limitarsi a):

- **output UX diretti e indiretti** — messaggi, copy, micro-copy automatico, notifiche, schermate
- **suggerimenti impliciti** — hint, raccomandazioni, prompt non richiesti
- **indicatori visivi e segnali** — icone, badge, segnali di stato
- **timing percepibile** — ritardi intenzionali, “pause comunicative”, modifiche del ritmo o dell’aspettativa senza autorizzazione
- **placeholder intelligenti** — placeholder adattivi o contestuali non autorizzati
- **CTA latenti** — call-to-action non esplicitamente autorizzate
- **cambiamenti di comportamento osservabili** — qualsiasi modifica dell’esperienza percepita dall’utente senza autorizzazione

**Indipendenza:** I test di integrità UX sono **indipendenti** da tutti gli altri layer; **non** dipendono da:

- Behavioral Memory (6.2)
- Capability Registry (6.3)
- Micro-Payments / Value Infrastructure (6.4)
- Killer Features (6.5)
- Feature Tier Mapping, Kill-Switch, Futile Chat Detection (6.6.1)

I test **verificano** che questi layer **non** producano effetto osservabile in assenza di autorizzazione esplicita; il criterio di pass/fail **MUST** essere definito in modo indipendente dalla loro implementazione.

---

## 5. Tipologie di test concettuali

Le seguenti sottosezioni descrivono **cosa viene verificato** da ciascuna tipologia di test, **non** come implementarlo. Nessuno strumento, framework né pipeline di test è prescritto; la spec definisce **solo** requisiti architetturali verificabili.

### 5.1 Test di silenzio assoluto

**Requisito:** Verifica che, in **configurazione base** (nessuna feature autorizzata a produrre output):

- **nessun** messaggio venga mostrato
- **nessun** suggerimento venga proposto
- **nessun** hint o segnale UX sia presente

**Criterio di successo:** Il sistema **MUST** essere **completamente muto**. L’esperienza **MUST** essere indistinguibile da quella di un sistema senza output UX.

### 5.2 Test di regressione percettiva

**Requisito:** Verifica che l’introduzione o modifica di **qualsiasi** componente:

- **non** alteri l’esperienza percepita (in assenza di autorizzazione esplicita per tale alterazione)
- **non** introduca output indiretti
- **non** cambi timing, forma o aspettative dell’utente senza autorizzazione

**Criterio di successo:** Il risultato **percepito** prima e dopo la modifica **MUST** essere **indistinguibile** (per contesti in cui nessuna nuova autorizzazione è stata introdotta). La **regressione percettiva** **MUST** essere trattata come **bug critico**.

### 5.3 Test di isolamento dei layer

**Requisito:** Verifica che:

- **componenti shadow** — Behavioral Snapshot, Ledger Shadow, Value Shadow
- **listener passivi** — observer, subscriber che non producono output diretto
- **snapshot, ledger, registry** — strutture dati e layer 6.2, 6.3, 6.4

**non** producano **alcun effetto osservabile** in assenza di autorizzazione esplicita.

**Criterio di successo:** L’attivazione o presenza di questi componenti **MUST NOT** modificare l’esperienza percettiva dell’utente; **nessun** output **MUST** essere tracciabile a questi layer senza un’autorizzazione esplicita a valle.

### 5.4 Test di failure e degradazione (fail-silent)

**Requisito:** In caso di:

- **errore interno**
- **stato inconsistente**
- **configurazione incompleta**
- **eccezioni non gestite**

il sistema **MUST**:

- **fallire nel silenzio** — nessun output **MUST** essere prodotto; il comportamento **MUST** essere fall-back to silence (coerente con 6.6.2)
- **non** esporre errori all’utente (salvo contesti esplicitamente autorizzati per messaggi di errore)
- **non** produrre output compensativi — nessun messaggio di fallback, placeholder di errore o suggerimento “di recupero” non autorizzato

**Criterio di successo:** In condizione di failure, l’esperienza **MUST** restare **silente**; **nessun** output **MUST** essere introdotto come conseguenza dell’errore.

---

## 6. Comportamenti esplicitamente vietati (test-failing)

I test **MUST** **fallire** (risultato = **falliti**, non pass) se rilevano **qualsiasi** dei seguenti comportamenti:

- **output “neutri” ma visibili** — messaggi o segnali presentati come neutri ma comunque osservabili dall’utente senza autorizzazione
- **placeholder intelligenti** — placeholder adattivi, contestuali o “intelligenti” senza autorizzazione
- **micro-copy automatico** — testo generato automaticamente e mostrato senza autorizzazione esplicita
- **indicatori che rivelano attività IRIS** — qualsiasi indicazione visibile che il sistema sia “in ascolto”, “attivo” o “pronto” senza autorizzazione
- **messaggi che spiegano il silenzio** — spiegazioni del tipo “non sto intervenendo perché…” (violazione di 6.6.2)
- **latenze comunicative intenzionali** — ritardi, pause o indicatori di attesa usati per veicolare “qualcosa in corso” senza autorizzazione
- **suggerimenti “educativi”** — hint o raccomandazioni non richieste, anche se presentate come utili

**Questi comportamenti rendono i test falliti.** Qualsiasi output **osservabile** dall’utente **senza** autorizzazione esplicita e verificabile **MUST** far **fallire** i test di integrità UX; **nessuna** eccezione tacita è ammessa.

## 8. Posizionamento architetturale

**UX Integrity Tests**:

- **non** sono runtime — **non** partecipano al flusso applicativo in esecuzione
- **non** sono una feature — **non** aggiungono funzionalità all’utente
- **non** sono un servizio — **non** espongono capacità invocabili nel flusso normale

Sono un **guardrail trasversale alla Fase 6** (Event, Behavior & Value Layer). I test **verificano** che il sistema rispetti Silence-First e l’assenza di output non autorizzato; **non** producono output UX; **non** modificano il comportamento del sistema in produzione.

---

## 9. Evolvibilità controllata

**È consentito:**

- **aggiungere nuovi test di integrità** — estendere la copertura a nuovi tipi di output o layer, **senza** ridurre la severità dei criteri esistenti
- **raffinare il concetto di “output UX”** — chiarire i confini di ciò che è considerato output osservabile, **senza** introdurre eccezioni implicite

**È vietato:**

- **rendere opzionale l’integrità UX** — l’integrità percettiva **MUST** restare **obbligatoria**; **MUST NOT** essere disattivabile o degradabile
- **indebolire i test** — **MUST NOT** allargare i criteri di pass in modo da accettare output non autorizzati
- **introdurre eccezioni implicite** — **MUST NOT** definire casi in cui l’output “implicito” o “contestuale” sia accettabile senza autorizzazione esplicita
