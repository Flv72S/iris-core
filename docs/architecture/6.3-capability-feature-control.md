## 1. Scopo del Blocco 6.3

Il BLOCCO 6.3 è un **layer decisionale puro**: risponde alla domanda «una data capability o feature è attualmente abilitata o disattivata?» senza introdurre comportamento, UX o inferenza.

Si dichiara esplicitamente che il Blocco 6.3:

- **non introduce comportamento** — non esegue azioni, non modifica flussi applicativi, non invoca servizi di business
- **non introduce UX** — nessuna schermata, widget o flusso utente è definito o controllato da 6.3; il blocco è shadow
- **non introduce inferenza** — nessuna deduzione da dati, eventi o Behavioral Memory; nessun scoring, soglia o adattività

**Distinzione netta (vincolante):**

| Concetto | Definizione | Esempio |
|----------|-------------|---------|
| **Possibilità tecnica** | La capability *esiste* nel registro e può essere interrogata; stato iniziale OFF/SHADOW | «canSuggestSummary è definita» |
| **Autorizzazione** | Un tier o una configurazione *abilita* la capability per un contesto; il layer 6.3 non concede, solo *non nega* | «tier premium abilita canSuggestSummary» |
| **Disattivazione** | Un layer superiore (Kill-Switch) *nega* la capability indipendentemente da tier o configurazione | «Kill-Switch spegne canSuggestSummary globalmente» |

Il Blocco 6.3 **non concede** diritti; può solo **consultare** (registry, tier mapping) e **negare** (kill-switch). L’abilitazione è assenza di negazione + valore ammesso da tier/configurazione, non una decisione attiva del blocco.

---

## 2. Layering e Ordine di Precedenza (vincolante)

L’ordine tra i layer del Blocco 6.3 è **rigido** e **non invertibile**:

1. **Kill-Switch Globale (6.3.3)** — massima precedenza
2. **Feature Tier Mapping (6.3.2)** — precedenza intermedia
3. **Capability Registry (6.3.1)** — base dichiarativa

**Regola vincolante:**

> Un layer superiore può solo **negare**, mai **concedere**.

- Il **Kill-Switch (6.3.3)** può solo **spegnere** una capability; non può abilitarla.
- Il **Feature Tier Mapping (6.3.2)** determina quali capability sono *ammesse* per un tier; non introduce capacità nuove, solo mappatura deterministica tier → capability.
- Il **Capability Registry (6.3.1)** dichiara *quali* capability esistono e il loro stato di base (OFF/SHADOW); non decide per utenti o contesti.

L’esito finale «capability X abilitata?» è: **non negata da 6.3.3** ∧ **ammessa da 6.3.2 per il tier** ∧ **definita e non default-deny in 6.3.1**. Nessun layer può, da solo, «accendere» una capability che un layer superiore ha negato o che il registro non definisce come abilitabile.

## 3. Responsabilità dei Microstep

### 6.3.1 — Capability Registry

- **Registro dichiarativo** delle capability esistenti; elenco finito e tipizzato, stato booleano (abilitato/disabilitato).
- **Stato iniziale:** OFF o SHADOW per ogni capability; default-deny.
- **Nessuna decisione** su utenti, tier o rollout; il registry non sa «chi» o «quando», solo «quale capability esiste e qual è il valore di base».
- Riferimento: [6.3.1-capability-registry.md](./6.3.1-capability-registry.md).

---

### 6.3.2 — Feature Tier Mapping

- **Mappatura deterministica** tier → capability: per un dato tier, quali capability sono *ammesse* (non negate a questo livello).
- **Nessuna dipendenza** da eventi, Behavioral Snapshot o tempo; la mappatura è configurazione/tabella, non calcolo.
- **Nessuna inferenza o adattività**; nessun «se utente ha fatto X allora tier Y».
- Un layer superiore (6.3.3) può comunque **negare** qualsiasi capability indipendentemente dal tier.

---

### 6.3.3 — Kill-Switch Globale

- **Meccanismo sovrano di disattivazione**: può spegnere una o più capability (o l’intero blocco) indipendentemente da tier o registro.
- **Fail-safe**: in caso di errore o indisponibilità del meccanismo di kill-switch, il comportamento **MUST** essere conservativo (es. considerare «disattivato» dove ambiguo).
- **Può solo spegnere, mai accendere**; non concede capacità, solo nega.

## 4. Regole Architetturali Vincolanti

Le seguenti regole sono **non negoziabili** per il Blocco 6.3:

| Regola | Descrizione |
|--------|-------------|
| **Separazione capability / feature / UX** | Una *capability* è un permesso tecnico binario; una *feature* è un comportamento osservabile; l’*UX* è l’interfaccia. Il blocco 6.3 gestisce solo capability; non definisce feature né UX. |
| **Assenza totale di inferenza** | Nessuna deduzione da eventi, Behavioral Memory (6.2), dominio o contesto utente. Nessun scoring, soglia, ML o adattività. |
| **Invarianza del dominio** | Il **dominio** (use case, entità, repository) **non dipende** dal Blocco 6.3; non lo invoca, non ne legge. Con 6.3 presente o assente (NoOp), il dominio resta invariante. |
| **Shadow-first principle** | Tutto il Blocco 6.3 è **shadow**: non esposto all’utente, non parte dell’MVP visibile. Default è OFF/SHADOW; l’abilitazione è eccezione esplicita. |
| **Prevedibilità deterministica** | A parità di input (capability, tier, stato kill-switch), l’esito **MUST** essere identico. Nessuna casualità, nessun stato volatile non dichiarato. |

## 5. Anti-Pattern Vietati

Le seguenti pratiche **violano** l’architettura del Blocco 6.3 e **non sono ammesse**:

| Anti-pattern | Descrizione | Esempio vietato |
|--------------|-------------|------------------|
| **Capability che decidono** | Una capability o il registry che contengono logica di business, soglie o condizioni («se X allora abilita») | «canSuggestSummary = true se numero messaggi > N» |
| **Tier dinamici** | Tier derivati da eventi, snapshot o comportamento utente invece che da configurazione/tabella fissa | «tier = premium se L1 indica utente attivo» |
| **Kill-Switch come feature flag UX** | Usare il kill-switch per abilitare funzionalità visibili o per A/B test invece che per disattivazione di sicurezza/controllo | «Kill-Switch ON = mostra nuovo pulsante» |
| **Dipendenze da Behavioral Memory** | Il Blocco 6.3 legge L0/L1 (6.2.1, 6.2.2) o eventi (6.1) per decidere capability o tier | «capability X = true se snapshot L1 > soglia» |
| **Concessione da layer superiore** | Un layer (es. Kill-Switch) che «abilita» una capability invece che solo negarla | «Kill-Switch concede canSuggestSummary» |
| **Dominio che dipende da 6.3** | Use case o entità che invocano il Capability Registry o il Feature Tier Mapping per completare flussi di business | «Use case legge 6.3 per decidere se inviare messaggio» |

Qualsiasi violazione richiede **revisione formale** di questo documento.
