# ADR-0001 — CQRS Strict Separation

## Status
Accepted

## Context
Il sistema cresce in complessità. Letture e scritture hanno requisiti diversi: throughput, modelli, latenza. Accoppiare read e write nello stesso flusso produce side-effect involontari e rende difficile ragionare sui flussi di dati. Senza un confine esplicito, il codice tende a mescolare responsabilità e a degradare nel tempo.

## Decision
Separazione rigida Read Side / Write Side. Nessuna dipendenza incrociata tra i due lati. L’enforcement è automatico tramite regole statiche applicate al grafo degli import. CQRS è un vincolo strutturale del progetto, non un pattern opzionale.

## Consequences

### Positive
- Isolamento dei flussi: read e write evolvono senza interferire.
- Testabilità: i due lati si testano in modo indipendente.
- Prevedibilità: chi modifica il read non tocca il write e viceversa.

### Negative
- Duplicazione concettuale dove lo stesso concetto di dominio appare sia in read che in write.
- Maggiore disciplina: ogni nuovo modulo deve rispettare il confine.

### Trade-offs accettati
- Più file e più regole in cambio di prevedibilità e manutenibilità a lungo termine.
