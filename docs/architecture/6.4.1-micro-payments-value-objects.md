# 6.4.1 — Micro-Payments Foundation
## Value Objects — Specifica architetturale concettuale ufficiale (SPEC-ONLY)

**Microstep:** 6.4.1 — Micro-Payments Foundation (Value Objects)  
**Blocco:** 6.4 — Micro-Payments Foundation  
**Tipo:** Spec concettuale vincolante  
**Stato:** NON implementata  
**Ambito:** Infrastrutturale (no UX, no AI, no side-effect)  
**Prerequisiti:** Nessuna dipendenza da 6.2 (Behavioral Memory), 6.3 (Capability & Feature Control Layer)

Questo microstep definisce l’**insieme minimo, chiuso e deterministico** di Value Objects necessari a rappresentare micro-valore all’interno di IRIS. **Non abilita pagamenti**, **non trasferisce valore reale**, **non espone API**, **non modifica** alcun comportamento del sistema. Spec **puramente concettuale**; **nessuna implementazione** in questo microstep.

---

## 1. Scopo

Definire l’**insieme minimo, chiuso e deterministico** di Value Objects necessari a rappresentare micro-valore all’interno di IRIS, in modo:

- **indipendente da provider di pagamento** — nessun binding a Stripe, Apple, ecc.; nessuna conversione in valuta reale
- **privo di logica di business** — nessuna regola di spesa, credito, billing o pricing
- **privo di UX** — nessuna schermata, copy o affordance economica
- **privo di inferenze** — nessun ranking, scoring o interpretazione AI
- **completamente reversibile e auditabile** — Value Objects puri, immutabili, deterministici

Si dichiara esplicitamente che questo microstep **non abilita pagamenti**, **non trasferisce valore reale**, **non espone API**, **non modifica** alcun comportamento del sistema. È **completamente silente**.

---

## 2. Posizionamento architetturale

Il microstep 6.4.1 è **parte del BLOCCO 6.4 — Micro-Payments Foundation**.

**Dipende solo da:**

- primitive di dominio (identità, tempo) — dove strettamente necessarie per riferimenti o contesto; nessuna logica di business

**NON dipende da:**

- **Behavioral Memory (6.2)** — nessuna lettura di L0/L1
- **Capability Registry / Feature Tier Mapping / Kill-Switch (6.3)** — nessuna capability attivata, tier mapping irrilevante in 6.4.1
- **AI / inference / ranking** — nessuna deduzione, nessun scoring
- **UX o UI** — nessuna interfaccia, nessun flusso utente

I Value Objects definiti qui **saranno consumati passivamente** dai microstep successivi (6.4.2+). In 6.4.1 **nessun flusso di valore** è definito né attivato.

---

## 3. Principi fondamentali

Tutti i Value Objects definiti in questo documento **DEVONO** rispettare:

| Principio | Descrizione |
|-----------|-------------|
| **Immutabilità** | Nessuna modifica dopo la creazione; ogni “variazione” produce una nuova istanza |
| **Eguaglianza per valore** | Due istanze con stesso valore sono considerate equivalenti; nessuna identità di riferimento |
| **Assenza totale di side-effect** | Nessuna scrittura, nessuna chiamata a rete, nessuna modifica di stato esterno |
| **Assenza di logica decisionale** | Nessuna soglia, regola di business, autorizzazione o calcolo di prezzo |
| **Assenza di riferimenti a storage, rete o provider** | Nessun binding a database, API esterne o provider di pagamento |
| **Assenza di inferenza** | Nessuna deduzione, ranking, scoring o interpretazione da parte di AI o logica applicativa |
| **Assenza di conversioni automatiche** | Nessuna conversione in valuta reale, nessun tasso di cambio |
| **Assenza di stato derivato** | Nessun calcolo di saldo, credito o debito; solo rappresentazione di quantità e unità |

**Violazioni** di uno o più principi **invalidano** il microstep e **MUST** essere trattate come **nuovo microstep** (es. 6.4.2 o successivo), con documento e confini propri.

---

## 4. Value Objects definiti

### 4.1 MicroValue

Rappresenta una **quantità atomica di valore interno**.

**Caratteristiche concettuali:**

- **quantità numerica discreta** — intero non negativo; nessun decimale, nessuna frazione
- **unità astratta** — non valuta; non EUR, USD o altra valuta reale
- **non frazionabile** — quantità atomica; eventuale frazionamento è fuori ambito 6.4.1
- **non convertibile** — nessuna conversione in denaro reale o altra unità esterna

**MicroValue non rappresenta denaro reale.** È un valore interno, simbolico, utilizzabile solo nel perimetro definito da microstep successivi.

---

### 4.2 ValueUnit

**Identificatore simbolico** dell’unità di valore.

**Esempi concettuali (non vincolanti):**

- TOKEN
- CREDIT
- MICRO_POINT

**Vincoli:**

- L’unità **non ha** tasso di cambio
- L’unità **non ha** semantica economica (non è “euro”, “dollaro”, “punto fedeltà” con valore legale)
- L’unità **non ha** valore esterno (nessuna conversione in moneta reale)

ValueUnit è **solo** un nome simbolico per distinguere tipi di valore interno; non implica valore di mercato né diritti contrattuali.

---

### 4.3 ValueBalance

Rappresenta una **quantità di MicroValue associata a una singola unità** (ValueUnit).

**Regole vincolanti:**

- **sempre ≥ 0** — nessun saldo negativo; nessun “debito” o “credito” in senso contabile
- **non va mai in negativo** — nessuna operazione che porti sotto zero è definita in 6.4.1
- **non mutabile** — ogni variazione concettuale (es. in microstep successivi) **crea una nuova istanza**; l’istanza esistente resta immutabile

**Nessuna logica di “spesa” o “credito”** è definita in 6.4.1. ValueBalance è **solo** la rappresentazione di (quantità, unità); non contiene operazioni di prelievo o accredito.

---

### 4.4 ValueOwnerRef

**Riferimento astratto** al detentore del valore.

**Proprietà:**

- può rappresentare: **utente**, **sistema**, **entità tecnica** (identificatore simbolico)
- **non implica** identità reale (nessun binding a account di pagamento o identità federata)
- **non implica** diritti (nessuna autorizzazione, nessun diritto contrattuale)

**ValueOwnerRef è solo un identificatore**, non un account. Non contiene saldo, non espone operazioni, non è collegato a provider o storage.

---

### 4.5 ValueReason (opzionale, ma raccomandato)

**Etichetta semantica non computabile** che descrive il contesto del valore.

**Esempi concettuali (non vincolanti):**

- TIP
- BOOST
- THANK_YOU
- SYSTEM_GRANT

**Vincoli:**

- **Non influisce** su calcoli — nessun uso in formule, soglie o autorizzazioni
- **Non influisce** su autorizzazioni — nessun binding a capability o tier
- È **solo** contesto descrittivo per audit o tracciabilità; nessuna logica decisionale

---

## 5. Invarianti architetturali (vincolanti)

**Nessun Value Object definito in 6.4.1 può:**

- **accedere a dati esterni** — nessuna lettura da storage, rete o provider
- **calcolare prezzi** — nessun prezzo, nessun tasso, nessuna conversione
- **applicare regole di business** — nessuna logica di spesa, credito, billing o autorizzazione
- **mutare stato interno** — immutabilità obbligatoria

**Non esiste in 6.4.1:**

- **wallet** — nessun concetto di portafoglio o contenitore mutabile
- **conto** — nessun account con operazioni di addebito/accredito
- **saldo globale** — nessun aggregato di saldi gestito da questi Value Objects
- **trasferimento** — nessun flusso di valore tra entità; i Value Objects sono **solo** strutture di dati pure
- **autorizzazione** — nessuna decisione su “può spendere” o “può ricevere”
- **pricing** — nessun prezzo, nessun calcolo di costo
- **billing** — nessuna fatturazione, nessun addebito reale

Ogni tentativo di usare questi oggetti per **monetizzazione**, **billing** o **UX economica** è **fuori ambito** di 6.4.1 e **MUST** essere trattato come **nuovo microstep** (es. 6.4.2+).

---

## 6. Relazione con UX e AI

| Layer | Relazione con 6.4.1 |
|-------|----------------------|
| **UX** | **Non conosce** l’esistenza di questi oggetti; nessuna schermata, copy o affordance economica in 6.4.1 |
| **AI** | **Non legge**, **non interpreta**, **non suggerisce** in base a MicroValue, ValueBalance o ValueReason; nessun ranking o scoring |
| **Capability Registry (6.3)** | **Nessuna capability attivata** in 6.4.1; eventuale canSendMicroValue o simili è fuori ambito 6.4.1 |
| **Tier Mapping (6.3)** | **Irrilevante** per 6.4.1; i Value Objects non dipendono da tier |

Questo microstep è **completamente silente**: nessun effetto osservabile sul sistema, nessuna modifica di comportamento, nessuna esposizione a UX o AI.

---

## 7. Anti-pattern vietati

È **vietato** in 6.4.1:

| Anti-pattern | Descrizione |
|--------------|-------------|
| **Conversione in valuta** | Nessuna conversione MicroValue → EUR, USD o altra valuta reale |
| **Riferimenti a pagamenti reali** | Nessun binding a Stripe, Apple Pay, bonifico, carta; nessuna “transazione” reale |
| **Saldo mutabile** | Nessuna modifica in-place di ValueBalance; solo nuova istanza |
| **Logica di spesa** | Nessuna operazione “deduci”, “accredita”, “trasferisci” nei Value Objects di 6.4.1 |
| **Binding a provider** | Nessun riferimento a Stripe, Apple, PayPal, ecc. |
| **UX copy o affordance** | Nessun testo “Saldo”, “Crediti”, “Acquista” né pulsanti/flow economici in 6.4.1 |
| **Uso per ranking o scoring** | Nessun uso di MicroValue o ValueBalance per ordinare utenti, suggerire contenuti o calcolare punteggi |
| **Monetizzazione implicita** | Nessun uso di questi Value Objects per implicare valore economico reale, diritto di scambio o equivalenza con denaro |

Qualsiasi violazione **viola** questa specifica e **MUST** essere trattata come **nuovo microstep**.

---

## 8. Evolvibilità controllata

**Consentito (con revisione architetturale o nuovo microstep):**

- **aggiungere** nuovi Value Objects **puri** (immutabili, senza side-effect, senza logica decisionale)
- **aggiungere** nuove unità simboliche (ValueUnit) — senza tasso di cambio né semantica economica
- **estendere** le etichette semantiche (ValueReason) — senza uso in calcoli o autorizzazioni

**Non consentito senza nuova spec:**

- **retro-inserire logica** — nessuna logica di spesa, credito, billing o pricing nei Value Objects di 6.4.1
- **introdurre conversioni** — nessuna conversione in valuta reale né tassi di cambio
- **agganciare billing o pricing** — nessun collegamento a sistemi di fatturazione o prezzi reali

## 9. Definition of Conceptual Done — 6.4.1

Il microstep **6.4.1 è considerato concettualmente completato** quando:

- [ ] Esiste un **set chiuso** di Value Objects (MicroValue, ValueUnit, ValueBalance, ValueOwnerRef, ValueReason o equivalente) definito e vincolato da questo documento
- [ ] **Tutti** sono **puri**, **immutabili** e **deterministici**; nessun side-effect, nessuna logica decisionale
- [ ] **Non esiste** alcun **flusso di valore** (nessun trasferimento, nessuna spesa, nessun accredito) in 6.4.1
- [ ] **Non esiste** alcun **effetto osservabile** nel sistema (nessuna modifica di comportamento, nessuna UX, nessuna API esposta)
- [ ] Il documento è **implementabile** senza modificare UX o comportamento esistente; i Value Objects sono **solo** strutture di dati consumabili da microstep successivi

## 10. Clausola di sicurezza architetturale

Qualsiasi estensione che:

- **introduca trasferimenti** (flussi di valore tra entità)
- **abiliti spesa** (deduzione, prelievo, utilizzo di MicroValue/ValueBalance)
- **modifichi saldo** (mutazione in-place di ValueBalance o concetto di wallet/conto)
- **colleghi a pagamenti reali** (conversione in valuta, integrazione con provider, billing)

**NON è un’estensione di 6.4.1** — **richiede un nuovo microstep esplicito** (es. 6.4.2 o 6.4.3+), con documento e confini propri.

Il microstep 6.4.1 definisce **solo** Value Objects **puri**, **immutabili** e **deterministici** per la rappresentazione di micro-valore interno. Nessun trasferimento, nessuna spesa, nessun collegamento a denaro reale.

Questo documento è la **specifica architetturale concettuale ufficiale** del microstep 6.4.1; le implementazioni devono conformarvisi per essere considerate valide.
